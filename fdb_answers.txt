1. FoundationDB decouples TMS and SS to allow independent scaling of compute heavy transaction processing and capacity heavy storage workloads. TMS handles transaction coordination, concurrency control and write ahead logging to ensure ACID properties whereas the SS manages distribution of data, replication and serves read queries without being bottlenecked.
2. Read write conflicts are detected by the resolvers in the TMS. This conflict detection occurs exclusively during the commit phase of a transaction. The resolvers ensure serializability by checking if the read sets of the currently committing transaction intersect with the write sets of any other transactions that have committed since this transaction's initial read version.
3. The read version provides the transaction with an immutable, globally consistent snapshot of the database at that exact logical point in time. This mechanism enforces SSI by ensuring that all reads within the transaction see the exact same state without requiring read locks. Consequently, the transaction is completely isolated from any concurrent writes happening in the background.
4. Unlike standard reads, snapshot reads do not record their read ranges, meaning they completely bypass the resolvers' conflict detection phase during a commit. Because they do not register in the conflict checking process, they cannot cause concurrent write transactions to abort. You would choose a snapshot read for long running, read only tasks like background analytics, generating reports, or taking backups where strict serializability against concurrent writes is not required.
5. Recovery is exceptionally fast because FoundationDB does not rely on replaying a single, monolithic redo log upon failure. During normal operation, StorageServers proactively and continuously pull log mutations directly from the LogServers in real time, keeping their state nearly synchronized. When a failure occurs, the system simply stops the old transaction generation, identifies the latest consistent logical version, and spins up a new generation of coordinators, making recovery a metadata operation rather than a data replay operation.
